Figma designs, visible and trackable inside Jira.

Figma for Jira Data Center is a Jira DC plugin and AWS sync service that links Figma resources directly to Jira issues. Your team sees current design thumbnails, metadata, and status without leaving Jira. Built for Jira Data Center — not adapted from a Cloud workaround.

Jira Layer

Thin Jira DC plugin

A Java P2 plugin that acts as a proxy only. No business logic, no secrets stored in Jira.

Sync Layer

AWS sync service

3 Lambda functions, RDS PostgreSQL, S3 thumbnail cache, and Secrets Manager. Your infrastructure, your control.

Figma API

Webhooks + polling fallback

Change detection via Figma webhooks with a 30-minute polling fallback. No real-time dependency required.

What your team gains

01

Stop context switching

Designers and developers share a single source of truth for linked Figma resources, visible on every Jira issue. No tab-switching to check whether a design has changed.

02

Catch design changes early

When a linked file changes in Figma, the Jira issue updates automatically. No manual checks, no stale links discovered at review time.

03

Make handoff status explicit

Ready for Dev is a visible state in Jira, not a Slack message. When a design changes after that signal, status transitions to Changes After Dev automatically.

04

Full audit trail

Every link, status change, and sync event is recorded in the audit log. You have a traceable history of what changed and when — without relying on memory or Slack history.

What the plugin does

  • Link any Figma file, frame, or prototype to a Jira issue
  • View design thumbnail and metadata directly on the issue page
  • Detect file changes via Figma webhooks, with 30-minute polling fallback
  • Track design status: None, In Progress, Ready for Dev, Changes After Dev
  • Automatic status transition when a change is detected on a ready-to-build design
  • Audit log for all link and sync events
  • Admin-configurable via the Jira admin settings page

Jira Layer

Jira DC Plugin (Java P2)

Thin HTTP proxy. Routes requests to API Gateway. No stored secrets, no business logic.

API Layer

AWS API Gateway

x-api-key authentication. Routes to the appropriate Lambda function.

Compute

Lambda Functions

Three functions: api-handler (CRUD), webhook-handler (Figma events), polling-sync (30-min fallback).

Database

RDS PostgreSQL

Three tables: design_links, sync_state, audit_events. Deployed inside your VPC.

Storage

S3 Thumbnail Cache

Private bucket. Thumbnails are served via pre-signed URLs — never publicly accessible.

Secrets

AWS Secrets Manager

All credentials: Figma service token, database password, API key, webhook passcode. None hardcoded.

Secrets posture: All credentials are stored in AWS Secrets Manager. No secrets are hardcoded in source. The Jira plugin stores only the API Gateway URL and API key in Jira plugin settings — standard Jira configuration, not source code.

Common questions

We're on Jira Data Center, not Cloud — does this work for us?

Yes. This is a native Jira DC P2 plugin. It is built specifically for the Data Center deployment model and does not depend on Atlassian Cloud infrastructure or the Marketplace.

Is this production-ready?

It is a tested and documented starter package — TypeScript compiles clean, 44 unit tests pass, and the CDK stack synthesizes correctly. Production readiness requires your infrastructure setup: VPC/RDS, a Figma service token, and deployment to your AWS account. The package includes all documentation to guide that process.

Is this secure?

All credentials are stored in AWS Secrets Manager — no secrets are hardcoded anywhere. The S3 bucket is private; thumbnails are accessible only via short-lived pre-signed URLs. A SECURITY.md document covers the full credential storage model and security posture.

How does it connect to Figma without per-user authentication?

The integration uses a single org-level Figma service token, stored in AWS Secrets Manager. There is no per-user OAuth flow. This is appropriate for a single-organization internal deployment where access is controlled at the Jira level.

What is the maintenance overhead?

The sync service is Lambda-based — no servers to provision or patch. Sync runs automatically via Figma webhooks with a 30-minute polling fallback if webhooks are unavailable. Standard AWS CloudWatch logging is included for observability.
